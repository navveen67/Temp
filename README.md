As an architect, I see exactly why this is happening. This is a classic Hibernate behavior called **"Select-before-Insert."**

### Why it is happening:
Hibernate's `save()` method (from `JpaRepository`) must decide whether to perform an `INSERT` or an `UPDATE`. 
1. If the ID is generated by the database (like `@GeneratedValue`), Hibernate checks if the ID is null. If null, it inserts.
2. **Your Case:** You are using an `@EmbeddedId`. Because the ID is **manually assigned** (you set the `refId` and `requestType` before saving), Hibernate cannot know if this is a brand-new record or an existing one. 
3. To be safe, it executes a `SELECT` to see if that composite key already exists.

### The Problem in your Mapping:
Looking at your DB script, there is a mismatch:
*   **Database:** The Primary Key is actually a `bigserial` column named `id`.
*   **Entity:** The Primary Key is mapped as a composite `@EmbeddedId` (`ref_id`, `request_type`).

### The Solution: Use the `Persistable` Interface
To eliminate the `SELECT` call, you must tell Hibernate explicitly: *"Trust me, this is a new record, do not check the database."*

Here is the architectural fix:

#### 1. Update the Entity to implement `Persistable`
We will add a `@Transient` flag. When we create the entity, we set this flag to `true`. Hibernate will check this flag instead of calling the database.

```java
import org.springframework.data.domain.Persistable;
import javax.persistence.*;

@Entity
@Table(name = "tpt_limit")
@Data
public class TptLimitAudit implements Persistable<TptLimitAuditId> {

    @EmbeddedId
    private TptLimitAuditId id;

    // Map the actual DB primary key so Hibernate knows about it, 
    // but keep the composite key as the logical ID if required for your logic.
    @Column(name = "id", insertable = false, updatable = false)
    private Long internalId;

    @Column(name = "hash_cust_id", length = 64)
    private String hashCustId;

    // ... other columns ...

    @Transient
    private boolean isNew = true; // Default to true for new audit entries

    @Override
    public TptLimitAuditId getId() {
        return id;
    }

    @Override
    public boolean isNew() {
        return isNew;
    }

    // Use this lifecycle hook to ensure that after loading from DB, 
    // it's no longer considered "new"
    @PostLoad
    @PostPersist
    void markNotNew() {
        this.isNew = false;
    }
}
```

#### 2. Optimization for "Audit" Logic
Since this is an **Audit** table (`tpt_limit`), you are likely **only ever inserting** and never updating an existing audit row. 

If the database Primary Key is actually the `bigserial id`, the most efficient architectural move is to change the mapping to reflect the truth of the database:

**Recommended Corrected Mapping:**
```java
@Entity
@Table(name = "tpt_limit")
@Data
public class TptLimitAudit {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY) // Maps to bigserial
    private Long id;

    @Column(name = "ref_id")
    private String refId;

    @Column(name = "request_type")
    @Enumerated(EnumType.STRING)
    private RequestType requestType;

    // ... rest of the fields
}
```
**Why this is better:**
*   **Zero Selects:** Since `@Id` is now the auto-incrementing `id`, and it will be `null` when you call `repository.save()`, Hibernate knows it **must** be an insert. It will skip the `SELECT` entirely.
*   **Performance:** Composite keys are slower for indexing than a single `bigint`.
*   **Integrity:** You still have your `UNIQUE` constraint in the DB (`CONSTRAINT tpt_limit_ref_id_request_type_key`) which prevents duplicate data.
